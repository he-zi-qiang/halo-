数据模型 — 一切接口的根基

◆
为什么先看模型？ 接口本质是对数据的 CRUD。不理解表结构和关系，就无法构造有效的测试数据、验证返回值、预判边界条件。
P1 Schema 是什么？它在项目里做什么？
P1 prisma/schema.prisma 定义了所有表的结构
P1 6 个模型 → 6 张表，每个接口至少涉及一张
P2 User / Album / Song — 核心实体
P2 Playlist / Comment — 业务功能
P2 UserFavorite / PlaylistSong — 多对多关联表
P1 模型之间的关系对测试有什么影响？
P1 Song 必须属于 Album → 不能直接创建歌曲，必须先建专辑
P1 级联删除 → 删用户 = 删所有专辑+歌曲+评论+收藏
P2 唯一约束 → 测试边界的关键
P2 User.email 唯一 → 重复注册应返回错误
P2 [userId, songId] 唯一 → 不能重复收藏
P2 [playlistId, songId] 唯一 → 歌单不能加重复歌曲
P3 可选字段(?)与默认值 → 创建时可以省略哪些参数
prisma/schema.prisma 
// 核心关系链：User → Album → Song（级联删除）
model Song {
  id        Int      @id @default(autoincrement())
  title     String
  artist    String
  audioUrl  String
  duration  Float
  lyrics    String?       // ← 可选字段，测试时可以不传
  playCount Int      @default(0)  // ← 默认值，创建时自动为0
  albumId   Int            // ← 必须字段！Song 不能脱离 Album 存在
  album     Album    @relation(fields: [albumId], onDelete: Cascade)
}

model UserFavorite {
  userId  Int
  songId  Int
  @@unique([userId, songId])  // ← 联合唯一，重复收藏会触发 P2002 错误
}
⚠
测试启示：构造测试数据的顺序必须是 注册用户 → 创建专辑(含歌曲) → 然后才能测试歌曲相关接口（收藏、评论、歌单添加）。这就是数据依赖链。

P1 请求 URL 是怎么对应到代码的？
P1 app.js 注册前缀：app.use('/api/users', userRoutes)
P1 routes/user.js 定义具体路径：router.post('/login', ...)
P1 两段拼接 = 完整路径：/api/users + /login = /api/users/login
P1 HTTP 方法（GET/POST/PUT/DELETE）在哪里决定的？

P2 方法写错 = 404 → 常见的"明明路径对但就是不通"的原因

P2 没有 auth 的 = 公开接口 → router.get('/', songController.getSongs)

中间件 — 请求被处理前发生了什么

◆
中间件 = 管道上的阀门。 请求到达控制器之前，要经过一系列中间件。任何一个中间件拒绝，请求就到不了控制器。测试时遇到的 401、403、413 错误，基本都是中间件干的。
P1 auth 中间件做了什么？不带 token 会怎样？
P1 从 Header 提取 Authorization: Bearer <token>
P1 用 jwt.verify() 验证 → 失败返回 401
P1 成功 → 把 userId 挂到 req 上 → 控制器才能知道"谁在操作"
P2 Token 过期(7天) → 也是 401，但错误信息不同
P1 upload 中间件做了什么？
P1 解析 multipart/form-data → 把文件保存到 uploads/ 目录
P2 文件限制规则（测试边界的重点）
P2 cover: 仅 image/* 类型，最多 1 个
P2 songs: 仅 audio/* 类型，最多 20 个
P2 总大小: 200MB 上限 → 超过返回 413
P2 Express 内置中间件（app.js 里的那几行）
P2 express.json() → 解析 JSON 请求体 → 不设的话 req.body 是 undefined
P2 cors() → 允许跨域 → 不影响 Postman/curl 测试，影响浏览器
P3 express.static('/uploads') → 让上传文件可通过 URL 访问
middleware/auth.js 
const auth = (req, res, next) => {
  const authHeader = req.headers['authorization'];  // ← 取请求头
  const token = authHeader && authHeader.split(' ')[1]; // ← "Bearer xxx" 取后半段

  if (!token) return res.status(401).json({...}); // ← 没 token → 401

  jwt.verify(token, SECRET, (err, decoded) => {
    if (err) return res.status(401).json({...});   // ← token 无效 → 401
    req.userId = decoded.userId;  // ← 关键！后续控制器靠这个识别用户
    next();  // ← 放行，进入下一个中间件或控制器
  });
};

控制器 — 业务逻辑在这里

◆
控制器是测试的核心目标。 中间件只管"放不放行"，控制器决定"怎么处理数据"。大多数测试用例的设计依据，来自控制器源码中的 if/else 分支。
P1 注册/登录 — 如何设计测试用例？
P1 register — 看源码中的校验分支
P1 正常注册 → 201 + 返回 user + token
P1 缺少 email/username/password → 400
P2 邮箱已存在 → 400 "该邮箱已注册"
P1 login — 看源码中的错误路径
P1 正确邮箱+密码 → 200 + token
P1 邮箱不存在 → 401（通用错误信息）
P2 密码错误 → 也是 401 同样信息（安全设计：不暴露哪个错了）
P1 toggleFavorite — 一个接口两种行为
P1 未收藏 → 调用后收藏 → {favorited: true}
P1 已收藏 → 再调用取消 → {favorited: false}
P2 songId 不存在 → 外键约束报错
P2 createAlbum — 最复杂的接口（文件+事务）
P2 FormData 格式：cover(1) + songs(多个) + 文本字段
P2 Prisma 事务：Album + 多个 Song 原子创建
P3 music-metadata 提取音频时长 → 文件损坏时的处理
P2 getSongById — 有副作用的 GET 请求
P2 每次调用 playCount + 1 → 可以验证：调用前后 count 变化
P3 返回包含 top10 评论 + 收藏数 + 评论数
P2 权限校验 — 控制器里的"谁能操作"逻辑
P2 updateSong: album.userId === req.userId → 只有创建者能改歌词
P2 deleteAlbum: album.userId !== userId → 403
P2 getPlaylistById: playlist.userId !== userId → 403
controllers/userController.js → 如何从 if 推导测试用例 
async register(req, res, next) {
  const { email, username, password } = req.body;

  // 分支 1：缺少必填字段 → 测试用例：不传 email
  if (!email || !username || !password) {
    return res.status(400).json({ error: '请提供邮箱、用户名和密码' });
  }

  // 分支 2：邮箱重复 → 测试用例：用已注册邮箱再注册
  const existingUser = await prisma.user.findUnique({ where: { email } });
  if (existingUser) {
    return res.status(400).json({ error: '该邮箱已注册' });
  }

  // 分支 3：正常注册 → 测试用例：全部字段正确
  const hashedPassword = await bcrypt.hash(password, 10);
  const user = await prisma.user.create({...});
  const token = jwt.sign({userId: user.id}, SECRET, {expiresIn: '7d'});
  res.status(201).json({ user, token });  // ← 验证：status=201, 有 token
}
⚠
核心方法论：打开控制器源码 → 找到所有 if 分支 → 每个分支 = 一个测试用例。正常路径是一个用例，每个错误分支也是一个用例。这就是"从源码推导用例"。

测试实战 — 用工具验证你的理解

◆
前面 4 步是"读"源码，这一步是"动手"。 测试工具不重要，重要的是你知道要测什么、为什么测、期望什么结果——这些答案全在源码里。
P1 一个接口怎么测？完整步骤
P1 ① 读路由 → 确定 URL + 方法 + 是否要 auth
P1 ② 读控制器 → 确定请求参数(body/query/params)
P1 ③ 数 if 分支 → 得出测试用例数量
P1 ④ 准备依赖数据 → 根据 Schema 关系链构造
P1 ⑤ 发送请求 → 对比实际 vs 预期
P1 测试数据依赖链 — 按什么顺序准备数据？
P1 第一步：POST /api/users/register → 拿到 token
P1 第二步：POST /api/albums (带token+文件) → 拿到 albumId + songId
P1 第三步：现在才能测 收藏/评论/歌单/歌词更新…
P2 验证响应的哪些部分？
P1 Status Code — 200/201/400/401/403/404/500
P2 Response Body — 字段是否齐全、类型是否正确
P2 数据库状态 — 记录是否真的创建/删除/修改了
P3 副作用 — playCount 增长、文件是否写入磁盘

实战示例：测试"收藏切换"接口的完整流程

curl 测试流程 
# ① 准备数据：先登录拿 token（假设 seed 数据已存在）
curl -X POST http://localhost:3000/api/users/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"123456"}'
# → 拿到 token: "eyJhbGci..."

# ② 测试正常路径：收藏歌曲(songId=1)
curl -X POST http://localhost:3000/api/users/favorites \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGci..." \
  -d '{"songId": 1}'
# → 期望：200, {"message":"...", "favorited": true}

# ③ 测试切换：再次调用同一接口 → 应该取消收藏
curl -X POST http://localhost:3000/api/users/favorites \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGci..." \
  -d '{"songId": 1}'
# → 期望：200, {"message":"...", "favorited": false}

# ④ 测试无 token → 应该被 auth 中间件拦截
curl -X POST http://localhost:3000/api/users/favorites \
  -H "Content-Type: application/json" \
  -d '{"songId": 1}'
# → 期望：401

调试排错 — 测试不通过时怎么办

◆
状态码是最好的线索。 每个状态码都指向代码中特定的位置。看到错误码，就知道该去哪行代码找原因。
P1 收到异常状态码 → 去哪里找原因？
P1 401 → 中间件 auth.js
P2 没带 token？格式不对("Bearer "少空格)？过期了？
P1 400 → 控制器的参数校验
P2 看错误消息 → 定位到控制器中具体哪个 if
P1 404 → 路由不匹配 或 资源不存在
P2 "请求的资源不存在" → URL/方法写错了（app.js 兜底）
P2 业务 404 → 控制器里 findUnique 返回 null
P2 403 → 权限不足（控制器的 userId 检查）
P3 用 A 用户的 token 操作 B 用户的资源
P2 500 → 代码运行时错误（catch 块）
P2 看后端终端日志 → console.error 会打印具体错误
P3 常见：Prisma 外键约束失败、字段类型不匹配
P2 状态码对了但数据不对 → 怎么排查？
P2 检查控制器中 include/select → 是否遗漏了关联数据
P2 用 Prisma Studio (npx prisma studio) 直接看数据库
P3 检查数据格式转换 → 如 PlaylistSong 展平为 Song 数组

状态码 → 源码位置 速查表

状态码含义产生位置排查方向
200成功控制器正常路径检查返回数据是否完整
201创建成功register / createPlaylist检查数据库是否有新记录
400参数错误控制器 if(!xxx) 分支看 error 消息 → 定位哪个字段
401未认证middleware/auth.js检查 Authorization 头格式
403无权限控制器 userId 校验确认 token 对应的用户是资源所有者
404不存在app.js 兜底 / 控制器检查 URL 拼写、HTTP 方法、资源 ID
500服务器错误控制器 catch 块看后端终端 console.error 输出
⚠
最常见的坑：① 忘记带 Content-Type: application/json 导致 body 为空 ② Token 的 "Bearer " 后面少了空格 ③ 用 POST 去请求 GET 接口（反之亦然）④ 忘记数据依赖链，直接测下游接口

总结：一张图串起全部

Schema
→ 告诉你数据长什么样、有什么约束、依赖关系 → 决定测试数据怎么构造
Routes
→ 告诉你URL 是什么、用什么方法、要不要认证 → 决定请求怎么发
Middleware
→ 告诉你什么条件会被拦截 → 决定认证/上传类的边界测试
Controller
→ 告诉你业务逻辑怎么走、有多少分支 → 决定测试用例有多少个
Testing
→ 把上面的理解转化为具体的请求和断言 → 验证代码是否按预期运行
Debugging
→ 用状态码反向定位源码位置 → 快速找到问题根因
◆
记住这个公式： 源码中的每个 if 分支 = 一个测试用例。正常路径 + 所有错误分支 = 完整的测试覆盖。不需要额外的文档，源码本身就是最准确的测试设计依据。
