接口设计（API 契约）

Sprint 第2-3天
定义 RESTful API 规范，这是前后端协作的「契约」
需求文档 → 定义 URL / 方法 / 字段 → 接口文档（OpenAPI）
输入：需求文档
输出：OpenAPI 接口文档（URL + 请求格式 + 响应格式）

Halo 文章管理接口清单：
  • GET /posts — 获取文章列表
  • POST /posts — 创建文章草稿
  • PUT /posts/{name} — 更新文章
  • PUT /posts/{name}/publish — 发布
  • PUT /posts/{name}/unpublish — 取消发布
  • PUT /posts/{name}/recycle — 回收
测试同步：根据接口文档编写正向/反向/边界测试用例 + Postman Collection 骨架
关键输出：两种文档 — OpenAPI 自动文档（给 SDK 生成用）+ 测试用接口文档（给人看）

后端

03 后端开发

Sprint 第3-7天
将接口设计落地为代码，Halo 包含三层：
路由定义 → PostEndpoint.java
数据模型 → Post.java @Schema 注解
业务逻辑 → PostServiceImpl.java
安全过滤 → SecurityWebFilter（横切）
输入：接口文档（OpenAPI 规范）
输出：可运行的 API 服务 + 自动生成的 OpenAPI JSON

三层代码结构：
  • 路由层 PostEndpoint.java — .GET("/posts", this::listPost) 链式定义
  • 模型层 Post.java — @Schema(required, minLength=1) 同时驱动校验+文档+SDK
  • 业务层 PostServiceImpl.java — 实际的创建/发布/回收逻辑
横切机制：SecurityWebFilter 在所有请求前检查认证（JWT / Basic Auth）
关键模式：operationId("DraftPost") → SDK 函数名 apiClient.post.draftPost()

04 前端开发

Sprint 第4-8天（与后端并行）
基于 TypeScript SDK 开发，不用手写 HTTP 请求：
@Schema 注解 → OpenAPI JSON
↓
TS SDK 自动生成 → Vue 页面调用
输入：接口文档 + 自动生成的 TypeScript SDK
输出：前端页面（PostList.vue, PostEditor.vue）

前后端并行的前提：接口契约已在 Phase 2 确定
  • 后端未就绪 → 前端用 Mock 数据开发
  • 后端完成一个接口 → 前端切换到真实 API
Halo 独特之处：SDK 自动生成，参数类型、响应类型全部自动推断
apiClient.post.draftPost({ postRequest: {...} })

05 接口测试

第2天起写用例，第4天起执行
后端每完成一个接口，测试立即验证：
正向测试 完整参数 → 200
反向测试 空标题 → 400
边界测试 1字符标题 → ?
4层测试覆盖：
  • 路由层 — URL 能否到达处理函数
  • 认证层 — 未认证请求是否拦截 → 401
  • 业务层 — 业务逻辑是否正确
  • 数据层 — 数据库是否正确读写
Postman 5步链式测试：
登录 → 创建草稿 → 查询验证 → 发布文章 → 回收文章

联调集成

Sprint 第8-10天（与测试尾部重叠）
前后端「合体」— 确保页面与接口完美配合
前端页面 + 后端 API → 端到端验证
常见问题：
  • 字段名不一致（前端 postTitle vs 后端 title）
  • 数据类型不匹配（string vs number）
  • 认证方式差异（Session vs Basic Auth）
  • 分页参数不一致（page 从 0 还是 1 开始）
Halo 优势：TypeScript SDK 从 OpenAPI 自动生成，字段名和类型不可能出错
→ 这就是「Single Source of Truth」的力量

联调清单：
  • PostEditor 创建文章 → PostList 显示
  • 点击发布 → 状态变为「已发布」
  • 点击回收 → 移入回收站
  • 未登录 → 重定向登录页

部署上线（CI/CD）

Sprint 第11天
通过流水线自动部署，接口测试以自动化回归形式运行
编译 → 单元测试 → 部署 Staging → 接口回归测试 → 部署 Production
CI/CD 6阶段流水线：
  • Stage 1 gradle build — 编译（30s）
  • Stage 2 gradle test — 单元测试（2min）
  • Stage 3 docker deploy staging — 部署测试环境（3min）
  • Stage 4 newman run — 接口回归测试（5min）
  • Stage 5 E2E 冒烟测试（10min）
  • Stage 6 docker deploy production — 全部通过后上线
Newman：Postman 的命令行工具，在 CI/CD 中自动运行测试集
回归价值：如果有人不小心改了路由路径，接口测试会立即失败，阻止合并

运维监控

上线后持续运行
接口健康检查 — 每5分钟自动检测核心接口可用性
GET /posts <500ms + GET /health → UP → 失败 → 告警
只用 GET 读操作做健康检查，绝不用 POST/PUT/DELETE（会产生脏数据）

每5分钟检查：
  • GET /api/v1alpha1/posts → 响应 < 500ms
  • GET /api/v1alpha1/categories → 响应 < 300ms
  • GET /actuator/health → status: UP
失败时：发送告警（邮件/Slack/钉钉），记录故障详情，触发排查流程

一个字段如何贯穿整条链路

以「创建文章」的 title 字段为例

1
后端代码定义
@Schema(requiredMode = REQUIRED, minLength = 1) String title
Java 注解同时定义了：校验规则 + 文档信息 + SDK 类型
2
OpenAPI 文档自动生成
title: { type: string, required: true, minLength: 1 }
SpringDoc 编译时从注解提取，生成 JSON 规范
3
TypeScript SDK 自动生成
interface PostSpec { title: string; /* required */ }
openapi-generator 从 OpenAPI JSON 生成类型安全的 SDK
4
前端调用
apiClient.post.draftPost({ postRequest: { post: { spec: { title: "..." } } } })
IDE 自动提示 title 是必填，类型错误编译时就报错
5
接口测试验证
正向：title="测试文章" → 200 | 反向：title="" → 400 | 边界：title="X"(1字符) → 200
一个 @Schema 注解，驱动了整条链路的一致性
后端定义 → 文档自动生成 → SDK 自动生成 → 前端类型安全 → 测试有据可依
这就是「Single Source of Truth」
