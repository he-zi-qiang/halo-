01

需求分析

Sprint 第1天

产品经理提出需求，所有角色参与评审。以 Halo「文章管理」功能为例：

需求描述

  • 用户能创建文章草稿，填写标题、正文、分类、标签
  • 用户能发布 / 取消发布文章
  • 用户能回收（软删除）文章
  • 用户能评论文章，管理员能审批 / 回复评论
  • 访客能浏览文章列表和文章详情
测试左移：测试工程师此时就参与需求评审，提前思考：
· 标题是否有长度限制？空标题能否提交？
· 未登录能否创建文章？权限如何控制？
· 草稿 → 发布 → 回收 → 再发布，状态流转是否合理？
这些问题会在后续转化为测试用例。

产品经理

输出需求文档和原型图，定义功能边界和用户故事

后端开发

评估技术可行性，初步设计数据模型和 API 结构

测试工程师

识别测试要点，规划测试策略，准备测试环境

02

接口设计

Sprint 第2-3天

后端架构师根据需求，设计 RESTful API 规范。这是前后端协作的「契约」。

Halo 文章管理接口清单

GET /api/v1alpha1/posts 获取文章列表（分页）
POST /api/v1alpha1/posts 创建文章草稿
PUT /api/v1alpha1/posts/{name} 更新文章信息
PUT /api/v1alpha1/posts/{name}/content 更新文章正文
PUT /api/v1alpha1/posts/{name}/publish 发布文章
PUT /api/v1alpha1/posts/{name}/unpublish 取消发布
PUT /api/v1alpha1/posts/{name}/recycle 回收（软删除）
DELETE /api/v1alpha1/posts/{name}/content 删除文章内容

评论管理接口清单

GET /api/v1alpha1/comments 获取评论列表
POST /api/v1alpha1/comments 创建评论
POST /api/v1alpha1/comments/{name}/reply 回复评论
注意接口路径的两种写法：
· 设计阶段简写：/api/v1alpha1/posts — 接口文档和代码中的路径
· 实际部署全路径：/apis/api.console.halo.run/v1alpha1/posts — Halo 服务器实际接收的 URL
Halo 在部署时会给 Console API 加上 /apis/api.console.halo.run 前缀作为命名空间。
后文 Postman 测试使用的是实际部署全路径。
关键输出 — 两种接口文档：
1. OpenAPI 自动文档：Halo 后端通过 @Schema 注解 + SpringDoc，在编译时自动生成 OpenAPI JSON 规范，包含完整的字段类型、校验规则、枚举值。前端 SDK 从此文档生成。
2. 测试用接口文档：如 Halo V2.2 接口文档，是面向测试人员的使用说明，包含 Base URL、认证方式、业务场景示例，内容源自 OpenAPI 文档但经人工整理补充了测试指引。
测试同步工作：根据接口文档编写测试用例，不用等后端开发完成！
测试工程师在此阶段产出：
· 每个接口的正向 / 反向 / 边界测试用例
· 接口间参数依赖关系（如创建文章 → 发布文章的 name 传递链）
· Postman Collection 骨架（请求模板 + 预期断言）
03

后端开发

Sprint 第3-7天

后端开发者将接口设计落地为代码。Halo 后端包含三层接口代码和一层横切安全机制：

第一层：路由定义（PostEndpoint.java）

文件路径：application/src/main/java/run/halo/app/core/endpoint/console/PostEndpoint.java

Halo 使用 Spring WebFlux 的 RouterFunction 链式定义所有路由：

Java · PostEndpoint.java 
@Override
public RouterFunction<ServerResponse> endpoint() {
  var tag = "PostV1alpha1Console";
  return SpringdocRouteBuilder.route()
    // 获取文章列表
    .GET("/posts", this::listPost,
      builder -> builder.operationId("ListPosts")
        .tag(tag).description("List posts"))
    // 创建文章草稿
    .POST("/posts", this::draftPost,
      builder -> builder.operationId("DraftPost")
        .tag(tag).description("Draft a post")
        .requestBody(requestBodyBuilder()
          .implementation(PostRequest.class)))
    // 更新文章
    .PUT("/posts/{name}", this::updatePost,
      builder -> builder.operationId("UpdatePost").tag(tag))
    // 发布文章
    .PUT("/posts/{name}/publish", this::publishPost,
      builder -> builder.operationId("PublishPost").tag(tag))
    // 取消发布
    .PUT("/posts/{name}/unpublish", this::unpublishPost,
      builder -> builder.operationId("UnpublishPost").tag(tag))
    // 回收文章
    .PUT("/posts/{name}/recycle", this::recyclePost,
      builder -> builder.operationId("RecyclePost").tag(tag))
    .build();
}
解读：每一行 .GET() / .POST() / .PUT() 定义一个接口。
operationId 会出现在自动生成的 API 文档中，前端 TypeScript SDK 据此生成函数名。
例如 operationId("DraftPost") → SDK 函数 apiClient.post.draftPost()

第二层：数据模型（Post.java — @Schema 注解）

Halo 用 Java 注解精确定义每个字段的规则，这些注解同时服务于三个目的：

代码校验

运行时验证请求数据合法性

文档生成

SpringDoc 据此生成 OpenAPI 文档

SDK 生成

前端 TypeScript SDK 据此生成类型

Java · Post.java PostSpec 
public class PostSpec {

  @Schema(requiredMode = RequiredMode.REQUIRED, minLength = 1)
  private String title;           // 必填，最少1字符

  @Schema(requiredMode = RequiredMode.REQUIRED, minLength = 1)
  private String slug;            // 必填，URL标识符

  @Schema(requiredMode = RequiredMode.REQUIRED)
  private Boolean deleted;        // 必填，是否已删除

  @Schema(requiredMode = RequiredMode.REQUIRED)
  private Boolean publish;        // 必填，是否发布

  @Schema(requiredMode = RequiredMode.REQUIRED)
  private Boolean pinned;         // 必填，是否置顶

  @Schema(requiredMode = RequiredMode.REQUIRED)
  private Boolean allowComment;   // 必填，是否允许评论

  @Schema(requiredMode = RequiredMode.REQUIRED)
  private VisibleEnum visible;   // 必填，枚举: PUBLIC | INTERNAL | PRIVATE

  @Schema(requiredMode = RequiredMode.REQUIRED)
  private Integer priority;       // 必填，排序优先级

  @Schema(requiredMode = RequiredMode.REQUIRED)
  private Excerpt excerpt;        // 必填，摘要配置
}
注解 → 文档 → 测试的对应关系：
代码注解文档显示测试用例
requiredMode = REQUIRED 字段标记为「必填」 缺少该字段时应返回 400
minLength = 1 最小长度 = 1 空字符串应返回 400
VisibleEnum 可选值：PUBLIC / INTERNAL / PRIVATE 传入 "INVALID" 应返回 400
defaultValue = "true" 默认值：true 不传时应默认为 true

第三层：请求体封装（PostRequest.java）

创建文章需同时提交「文章元数据」和「文章内容」，用一个 record 类封装：

Java · PostRequest.java 
public record PostRequest(
    @NonNull
    @Schema(requiredMode = RequiredMode.REQUIRED)
    Post post,          // 文章元数据（标题、slug、状态…）

    @NonNull
    @Schema(requiredMode = RequiredMode.REQUIRED)
    Content content      // 文章正文（raw + HTML + rawType）
) {}

两层 @NonNull + @Schema(REQUIRED) 双重保障，缺少任一字段都会被拒绝。

横切层：认证与安全（中间件 / 过滤器）

认证不属于某个具体接口的代码，而是作为全局中间件统一拦截所有请求。Halo 支持三种认证方式：

Basic Auth

用户名密码 Base64 编码
Authorization: Basic bWV0...
适合：Postman 调试、快速测试

Session + CSRF

登录获取 Session Cookie
写操作需带 CSRF Token
适合：浏览器前端（Halo Console）

PAT 个人令牌

Personal Access Token
Authorization: Bearer pat_...
适合：第三方集成、自动化脚本

HTTP · Session + CSRF 认证流程 
// 步骤1: 登录获取 Session 和 CSRF Token
POST /login
Content-Type: application/x-www-form-urlencoded

username=meterspheretest&password=Fit2cloud@2023

// 响应 Set-Cookie:
//   SESSION=abc123...
//   XSRF-TOKEN=xyz789...

// 步骤2: 后续写操作必须带上两个值
POST /api/v1alpha1/posts
Cookie: SESSION=abc123...
X-XSRF-TOKEN: xyz789...
Content-Type: application/json

{ ... }
后端开发完成的标志：
· 接口可以通过 Postman 独立调用并返回正确结果
· 返回值格式符合接口文档约定
· 认证机制正常工作（有凭证 → 通过，无凭证 → 401）
· 数据正确写入数据库
此时前端可能还没开发完！但接口已经可以被测试。
04

前端开发

Sprint 第4-8天（与后端并行推进）

前端不需要等后端全部完成才开始。接口设计（Phase 2）确定了 API 契约后，前端就能基于 TypeScript SDK 或接口文档进行开发。

前后端并行的前提：接口契约（URL、请求格式、响应格式）已在 Phase 2 确定。
· 后端未就绪时：前端用 SDK 类型定义编写代码，用 Mock 数据渲染页面
· 后端完成一个接口后：前端切换到真实 API 调用，逐步联调验证

Halo 前端的独特之处：TypeScript SDK 自动生成

后端 @Schema 注解
→
SpringDoc 生成 OpenAPI JSON
→
openapi-generator 生成 TS SDK
→
前端直接调用类型安全函数

前端不用手写 HTTP 请求代码，直接导入 SDK 函数即可。参数类型、响应类型全部自动推断。

// 创建草稿 — 对应后端 POST /api/v1alpha1/posts
const { data: newPost } = await apiClient.post.draftPost({
  postRequest: {

      spec: {

        priority: 0,
        excerpt: { autoGenerate: true }

    },
    content: {
      raw: formState.content,
      content: formState.contentHtml,
      rawType: "HTML"
    }
  }

// 保存返回的 name，后续操作都要用它

// 例如: "5152aea5-c2e8-4717-8bab-3e32db193457"

// 更新文章内容

  name: postName,

接口参数依赖链（关键概念）：
创建文章 → 返回 metadata.name → 用于发布 / 更新 / 回收
创建评论 → 返回 metadata.name → 用于回复 / 审批 / 删除

测试时也必须遵循这个顺序，Postman 用环境变量在接口间传递参数。
05

接口测试

Sprint 第2天起编写用例，第4-9天执行测试

质量保障的核心阶段。测试工作分两步：先根据接口文档编写用例（第2-3天，与后端开发并行），后端每完成一个接口再执行验证（第4天起）。

接口测试在测试金字塔中的位置

E2E 端到端测试（最少 · 最慢 · 最贵）
Selenium / Playwright 模拟浏览器操作
接口测试 API Test（核心主力 · 占 60-70%）
Postman / JUnit / pytest 直接调用接口
单元测试 Unit Test（最多 · 最快 · 最便宜）
JUnit / Jest 测试单个函数逻辑

接口测试的4层覆盖

测试层验证内容Halo 示例
路由层 URL 能否正确到达处理函数 GET /api/v1alpha1/posts 返回 200
认证层 未认证请求是否被拦截 不带 Authorization → 返回 401
业务层 业务逻辑是否正确 创建文章 → 返回 metadata.name
数据层 数据库是否正确读写 创建后 GET 能查到该文章

测试用例设计：正向 / 反向 / 边界

正向测试

  • 完整参数创建文章 → 200
  • 发布草稿 → publish=true
  • 分页查询第2页 → 正确数据
  • 回收文章 → deleted=true

反向测试

  • title 为空 → 400
  • visible 传 "INVALID" → 400
  • 不带认证 → 401
  • 操作别人的文章 → 403

边界测试

  • title 仅1字符 → 成功
  • title 超长10000字符 → ?
  • slug 含特殊字符 → ?
  • priority 为负数 → ?

为什么是「链式」？

06

联调集成

Sprint 第8-10天（与接口测试尾部重叠）

前后端代码基本完成，现在要「合体」— 确保前端页面与后端接口完美配合。

时间关系说明：联调并非等接口测试全部完成后才开始。
当后端核心接口通过接口测试后，前端就可以开始联调。此时接口测试的反向 / 边界用例可能仍在执行中。
两者并行推进：联调发现的问题也会补充到接口测试用例中。

联调中常见的问题

问题类型示例如何在接口测试阶段提前发现
字段名不一致 前端期望 postTitle，后端返回 title 接口文档 + SDK 自动生成可避免
数据类型不匹配 前端期望 string，后端返回 number Postman Tests 脚本验证响应格式
认证方式差异 前端用 Session，测试用 Basic Auth 同时覆盖两种认证方式的测试
分页参数不一致 前端从0开始计页，后端从1开始 接口测试验证边界页码（page=0）
Halo 的优势 — 契约驱动开发：
因为前端使用从 OpenAPI 自动生成的 TypeScript SDK，字段名和类型不可能出错 — 它们直接来自后端的 @Schema 注解。这就是「Single Source of Truth（单一数据源）」的力量。

联调验证清单

  • PostEditor.vue 创建文章 → 文章出现在 PostList.vue 列表中
  • PostList.vue 点击发布 → 文章状态变为「已发布」
  • PostList.vue 点击回收 → 文章移入回收站
  • PostEditor.vue 加载已有文章 → 内容完整显示（fetchPostHeadContent）

回归测试的价值 — 真实案例

假设某天开发者不小心修改了 PostEndpoint.java 中的路由：

// 原始代码
.PUT("/posts/{name}/publish", ...)

// 不小心改成了
.PUT("/posts/{name}/pub", ...)   // ← 路径变了！

CI/CD 中的接口测试会立即失败：

✗ FAIL  PUT /posts/{{postName}}/publish → 404 Not Found
        Expected status: 200, Received: 404

Pipeline FAILED — 代码被阻止合并到主分支

问题在部署之前就被捕获，不会影响线上用户。这就是自动化回归测试的核心价值。

全景回顾：一个字段如何贯穿整条链路

以 Halo「创建文章」的 title 字段为例

后端代码定义
@Schema(requiredMode = REQUIRED, minLength = 1) String title
↓
OpenAPI 文档自动生成
title: { type: string, required: true, minLength: 1 }
↓
TypeScript SDK 自动生成
interface PostSpec { title: string; /* required */ }
↓
前端调用
apiClient.post.draftPost({ postRequest: { post: { spec: { title: "..." } } } })
↓
接口测试验证
正向：title="测试" → 200 ✓  |  反向：title="" → 400 ✓  |  边界：title=1字符 → 200 ✓
一个 @Schema 注解，驱动了整条链路的一致性。
后端定义 → 文档自动生成 → SDK 自动生成 → 前端类型安全 → 测试用例有据可依
这就是现代软件工程中「单一数据源 Single Source of Truth」的核心实践。
