1什么是接口？

1.1 最直白的理解

接口（API，Application Programming Interface）就是前端和后端之间的「约定好的对话方式」。

想象你去餐厅吃饭：

  • 你（前端）：看菜单 → 跟服务员说「我要一份宫保鸡丁」
  • 服务员（接口）：把你的话翻译成厨房能懂的指令
  • 厨房（后端 + 数据库）：做菜 → 端给你

在这个过程中，你不需要知道厨房怎么做菜，厨房也不需要知道你坐在哪一桌。你们之间通过「菜单 + 服务员」这个接口来交流。

1.2 在 Halo 里，接口长什么样？

当你在 Halo 博客后台点击「发布文章」按钮时，实际发生的事是：

你在浏览器点击「发布」 │ ▼ 前端 Vue 代码执行： consoleApiClient.content.post.publishPost({ name: "文章ID" }) │ │ ── 自动组装成 HTTP 请求 ──→ │ ▼ 后端 Java 代码接收： PUT /apis/api.console.halo.run/v1alpha1/posts/{name}/publish 请求头：Authorization: Basic bWV0ZXJzcGhlcmV0ZXN0OkZpdDJjbG91ZEAyMDIz │ ▼ 后端处理逻辑： ① 验证身份（认证中间件） ② 查找文章（数据库查询） ③ 修改状态 phase: "DRAFT" → "PUBLISHED" ④ 保存到数据库 ⑤ 返回 JSON 结果 │ ▼ 前端收到响应： { "status": { "phase": "PUBLISHED" }, "spec": { "title": "Hello Halo" } } │ ▼ 页面更新：文章状态变为「已发布」

1.3 接口的本质：HTTP 请求 + JSON 响应

每一个接口调用，本质上就是一次 HTTP 通信，由以下 4 个部分组成：

组成部分说明Halo 实际例子
请求方法 你要做什么操作 GET 查看、POST 创建、PUT 修改、DELETE 删除
请求路径 你要操作哪个资源 /apis/api.console.halo.run/v1alpha1/posts
请求参数 操作的具体内容 URL 参数：?page=1&size=10
Body 参数：{"post":{...}, "content":{...}}
响应结果 后端返回的数据 状态码 200 + JSON 数据 {"items":[...],"total":5}
💡
RESTful 风格：Halo 的接口文档第一行就写了「接口类型：RESTful」。这是一种接口设计规范——用 HTTP 方法（GET/POST/PUT/DELETE）表达操作类型，用 URL 路径表达资源，用 JSON 传递数据。Halo 的所有接口都遵循这个规范。

2什么是接口测试？

2.1 先看完整系统有几层

一个 Web 应用从用户点击到数据库存储，经过 4 层：

层级 Halo 中对应的文件 / 组件 职责 ───────────────────────────────────────────────────────────── ① 前端 UI 层 PostList.vue / PostEditor.vue 按钮、表单、页面渲染 │ │ HTTP 请求 ▼ ② 接口路由层 PostEndpoint.java endpoint() 方法 URL 匹配、参数解析 │ ▼ ③ 认证 & 业务层 Basic Auth 验证 + PostService 身份验证、业务规则 │ ▼ ④ 数据层 Spring Data + 数据库 增删改查

2.2 接口测试覆盖哪些层？

接口测试 = 跳过第 ① 层（前端 UI），直接从第 ② 层开始，一直验证到第 ④ 层。

你用 Postman 或脚本代替了浏览器，直接向后端发 HTTP 请求：

┌──────────────────────────┐ 正常用户操作： │ │ 浏览器 → Vue 前端 → HTTP 请求 → 后端 │ ★ 接口测试在这里切入 ★ │ │ Postman 直接发 HTTP 请求 │ 接口测试： │ 跳过前端，直接对话后端 │ Postman ─────────→ HTTP 请求 → 后端 │ │ └──────────────────────────┘ 覆盖范围： ┌─────────────────────────────────────────────────────┐ │ ✅ 路由匹配 URL 对不对？方法是 GET 还是 POST？ │ │ ✅ 认证拦截 没带 token 会不会被拦截？ │ │ ✅ 参数校验 必填字段没填会不会报错？ │ │ ✅ 业务逻辑 数据处理规则对不对？ │ │ ✅ 数据库操作 数据真的存进去/取出来了吗？ │ │ ✅ 错误处理 异常情况返回的状态码和消息对不对？ │ └─────────────────────────────────────────────────────┘

2.3 为什么需要接口测试？——三个真实场景

1

场景一：前后端分离开发

Halo 的前端（Vue）和后端（Java）由不同的人开发。后端写好「创建文章接口」后，不需要等前端页面做好，就能用 Postman 验证接口是否正确。前端开发者也能通过接口文档提前知道要传什么参数、会收到什么返回。

2

场景二：精确定位 Bug

用户报告「发布文章失败」。是前端按钮没发请求？还是后端逻辑有问题？直接用 Postman 调用 PUT /posts/{name}/publish——如果 Postman 也失败，说明是后端 Bug；如果 Postman 成功，说明是前端 Bug。一步就缩小了排查范围。

3

场景三：自动化回归测试

Halo 仓库里有 PostEndpointTest.java 文件，用代码自动对每个接口发请求并验证结果。每次修改代码后自动运行，确保没有改坏已有功能。

2.4 测试金字塔——接口测试的投入产出比最高

E2E 测试（少量）
★ 接口测试（中量）—— 你正在学的
单元测试（大量）
测试类型覆盖范围速度定位精度成本
E2E 测试前端 → 后端 → 数据库（全部）最慢低高
接口测试路由 → 业务 → 数据库中等高低
单元测试单个函数最快最高最低

3什么是接口文档？

3.1 定义

接口文档是前后端之间的「合同」——它详细规定了每个接口的 URL 是什么、需要传什么参数、会返回什么数据。有了这份「合同」，前端和后端可以各自独立开发，最后对接时只要双方都按合同执行即可。

4后端如何创建（定义）一个接口？

4.1 Halo 后端的接口定义——源码剖析

Halo 的每一组接口都定义在一个 Endpoint 类中。以「文章接口」为例：

PostEndpoint.java —— 文章相关的全部接口

// 文件位置: application/src/main/java/run/halo/app/core/endpoint/console/PostEndpoint.java

@Slf4j                        // 日志功能
@Component                    // 注册为 Spring 组件（框架自动发现）
@RequiredArgsConstructor      // 自动注入依赖
public class PostEndpoint implements CustomEndpoint {

    private final PostService postService;       // 业务逻辑服务
    private final ReactiveExtensionClient client; // 数据库操作客户端

    @Override
    public RouterFunction<ServerResponse> endpoint() {
        return SpringdocRouteBuilder.route()

            // ═══ 查询类接口 (GET) ═══
            .GET("posts",
                 this::listPost,                    // 处理函数
                 builder -> builder
                    .operationId("ListPosts")        // API 文档：操作 ID
                    .description("List posts")       // API 文档：描述
                    .tag("PostV1alpha1Console")       // API 文档：分组标签
            )
            .GET("posts/{name}/head-content",  this::fetchHeadContent, ...)
            .GET("posts/{name}/release-content", this::fetchReleaseContent, ...)

            // ═══ 创建类接口 (POST) ═══
            .POST("posts",
                  this::draftPost,
                  builder -> builder
                    .operationId("DraftPost")
                    .requestBody(requestBodyBuilder()
                        .implementation(PostRequest.class))  // 请求体类型
            )

            // ═══ 修改类接口 (PUT) ═══
            .PUT("posts/{name}",          this::updatePost, ...)
            .PUT("posts/{name}/publish",   this::publishPost, ...)
            .PUT("posts/{name}/unpublish", this::unpublishPost, ...)
            .PUT("posts/{name}/recycle",   this::recyclePost, ...)

            // ═══ 删除类接口 (DELETE) ═══
            .DELETE("posts/{name}/content", this::deleteContent, ...)

            .build();
    }
}

4.3 后端处理函数做了什么？

每个路由后面的 this::draftPost 就是真正处理请求的方法。以创建文章为例：

// 简化后的处理逻辑
private Mono<ServerResponse> draftPost(ServerRequest request) {
    return request.bodyToMono(PostRequest.class)  // ① 解析请求体为 Java 对象
        .flatMap(postService::draftPost)            // ② 调用业务服务处理
        .flatMap(post -> ServerResponse              // ③ 构建响应
            .ok()                                    //    状态码 200
            .bodyValue(post)                         //    返回文章数据
        );
}

// PostService.draftPost() 内部做的事：
// ① 校验标题、slug 等必填字段
// ② 创建内容快照（Snapshot）
// ③ 创建文章记录（Post），关联快照
// ④ 返回创建好的文章对象

6认证是怎么做的？

Halo 接口文档 V2.2 详细介绍了三种认证方式。认证的核心问题是：后端怎么知道「你是谁」以及「你有没有权限」？

6.1 方式一：Basic Auth（基本认证）—— 最简单

🔑
文档原文：「通过简单设置 HTTP 请求头 Authorization: Basic [base64编码] 即可实现认证」
原理： 用户名: meterspheretest 密码: Fit2cloud@2023 ↓ 拼接为: meterspheretest:Fit2cloud@2023 ↓ Base64 编码: bWV0ZXJzcGhlcmV0ZXN0OkZpdDJjbG91ZEAyMDIz ↓ 放入请求头: Authorization: Basic bWV0ZXJzcGhlcmV0ZXN0OkZpdDJjbG91ZEAyMDIz ↓ 后端收到后： ① Base64 解码 → 得到 meterspheretest:Fit2cloud@2023 ② 拆分用户名和密码 ③ 验证密码是否正确 ④ 正确 → 放行；错误 → 返回 401 Unauthorized

生成 Base64 编码的方法（文档中提到的）：

# Linux / macOS 终端：
echo -n "meterspheretest:Fit2cloud@2023" | base64
→ bWV0ZXJzcGhlcmV0ZXN0OkZpdDJjbG91ZEAyMDIz

# Windows：访问 https://www.bejson.com/enc/base64 在线生成
⚠️
Basic Auth 的特点：每次请求都要带用户名密码（虽然是编码过的），适合简单场景和接口测试。Halo 接口文档中所有的 API 示例都使用了这种方式。

6.2 方式二：表单认证（Session）—— 模拟浏览器登录

这种方式模拟了用户在浏览器中填写登录表单的过程，分为三步：

1

获取 CSRF 令牌

GET http://halo.edu.metersphere.com/login

→ 返回 HTML 页面，其中包含：
<input type="hidden" name="_csrf" value="fff63640-029d-4857-9312-29e93229d6e0">

→ 响应头中包含：
Cookie: XSRF-TOKEN=fff63640-029d-4857-9312-29e93229d6e0

CSRF 令牌用于防止跨站请求伪造攻击——确保请求确实来自 Halo 自己的页面。

2

发送登录请求

POST http://halo.edu.metersphere.com/login
Content-Type: application/x-www-form-urlencoded
Cookie: XSRF-TOKEN=fff63640-029d-4857-9312-29e93229d6e0

_csrf=fff63640-029d-4857-9312-29e93229d6e0
&username=meterspheretest
&password=Fit2cloud@2023

→ 返回 200 + 响应头中：
Set-Cookie: SESSION=d04db9f7-d2a6-4b7c-9845-ef790eb4a980
3

后续请求携带 Session Cookie

GET /apis/api.console.halo.run/v1alpha1/posts
Cookie: SESSION=d04db9f7-d2a6-4b7c-9845-ef790eb4a980

→ 后端通过 Session ID 识别出用户身份，不需要再带用户名密码

6.3 方式三：MeterSphere 全局认证

这是在 MeterSphere 测试平台中使用的方式——在环境配置中选择 Basic Auth，填入用户名和密码，平台会自动为每个请求加上 Authorization 头，不需要手动填写。

6.4 三种认证方式对比

维度Basic Auth表单认证（Session）MeterSphere 认证
复杂度★☆☆ 最简单★★★ 最复杂（3 步）★☆☆ 配置后自动
适用场景接口测试、脚本调用模拟真实浏览器登录自动化测试平台
每次请求带什么Authorization: Basic xxxCookie: SESSION=xxx平台自动注入
安全性低（密码每次传输）高（只传 Session ID）取决于配置
Postman 中使用直接加 Header需要先调登录接口不适用
💡
Halo 文档明确推荐：「以下接口采用基本认证调用」。所以在 Postman 中测试 Halo 接口时，直接使用 Basic Auth 是最方便的。

8完整链路串讲：从写文章到发布的全部接口调用

下面用 Halo 的「创建文章 → 发布 → 评论 → 审核 → 回收」这个完整业务流程，串起前面学到的所有知识：

步骤 1：创建文章草稿 ══════════════════════════════════════════════════════════════════ 前端代码: consoleApiClient.content.post.draftPost({...}) 实际请求: POST /apis/api.console.halo.run/v1alpha1/posts 后端处理: PostEndpoint.draftPost() → PostService.draftPost() 数据库: 创建 Post 记录 (phase: DRAFT) 返回结果: { metadata: { name: "abc-123" }, status: { phase: "DRAFT" } } 文档章节: 3.6 │ │ 拿到 name = "abc-123" ▼ 步骤 2：发布文章 ══════════════════════════════════════════════════════════════════ 前端代码: consoleApiClient.content.post.publishPost({ name: "abc-123" }) 实际请求: PUT /apis/.../posts/abc-123/publish 后端处理: PostEndpoint.publishPost() → 修改 publish=true 数据库: 更新 Post 记录 (phase: DRAFT → PUBLISHED) 返回结果: { status: { phase: "PUBLISHED" }, spec: { publishTime: "..." } } 文档章节: 3.7 │ │ 文章已上线，用户可以评论 ▼ 步骤 3：发表评论 ══════════════════════════════════════════════════════════════════ 前端代码: consoleApiClient.comment.createComment({...}) 实际请求: POST /apis/.../comments (body 中 subjectRef.name = "abc-123") 后端处理: CommentEndpoint.createComment() 数据库: 创建 Comment 记录 (approved: false) 返回结果: { metadata: { name: "comment-456" }, spec: { approved: false } } 文档章节: 3.9 │ │ 拿到评论 name = "comment-456" ▼ 步骤 4：审核评论 ══════════════════════════════════════════════════════════════════ 实际请求: PUT /apis/.../comments/comment-456 (body 中 approved: true) 后端处理: 更新 approved: false → true 返回结果: { spec: { approved: true } } 文档章节: 3.10 │ ▼ 步骤 5：回收文章（软删除） ══════════════════════════════════════════════════════════════════ 前端代码: consoleApiClient.content.post.recyclePost({ name: "abc-123" }) 实际请求: PUT /apis/.../posts/abc-123/recycle 后端处理: PostEndpoint.recyclePost() → 修改 deleted=true 数据库: 更新 Post 记录 (deleted: false → true) 返回结果: { spec: { deleted: true } } 文档章节: 3.8
🎯
看到规律了吗？每一步都是同样的模式：
  1. 前端调用 SDK 方法（或你用 Postman 手动发请求）
  2. 变成一个 HTTP 请求（方法 + URL + 参数）
  3. 后端 Endpoint 匹配路由，调用处理方法
  4. 处理方法操作数据库，返回 JSON
  5. 前端（或 Postman）收到响应，从中提取 name 等关键字段，供下一个接口使用

这就是接口的全部。掌握了这个模式，任何项目的接口你都能看懂、能测、能调。

8.1 Postman 中测试这个完整流程的技巧

在 Postman 中，你可以创建一个 Collection（集合），按顺序排列这 5 个请求：

Postman Collection: "Halo 文章完整流程测试" ═══════════════════════════════════════════ 📁 1. 创建文章 (POST) → Tests 脚本中提取 name: pm.environment.set("postName", jsonData.metadata.name) 📁 2. 发布文章 (PUT) → URL 中使用变量: /posts/{{postName}}/publish 📁 3. 发表评论 (POST) → Body 中使用变量: "name": "{{postName}}" → Tests 中提取: pm.environment.set("commentName", jsonData.metadata.name) 📁 4. 审核评论 (PUT) → URL 中使用变量: /comments/{{commentName}} 📁 5. 回收文章 (PUT) → URL 中使用变量: /posts/{{postName}}/recycle 点击 "Run Collection" → 5 个请求按顺序自动执行 → 完整流程一键测完！
