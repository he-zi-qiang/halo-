01 什么是接口？

接口就是前端和后端之间约定好的对话方式。

👤
你（前端）
看菜单、点菜
「我要宫保鸡丁」
🤵

翻译指令、传递
「一份宫保鸡丁」
👨‍🍳
厨房（后端）
做菜、上菜

在 Halo 中，当你点击「发布文章」时，就是一次接口调用：

完整流程

每次接口调用由 4 个部分 组成：

请求方法

GET 查看 · POST 创建 · PUT 修改 · DELETE 删除

请求路径

/apis/api.console.halo.run/v1alpha1/posts

请求头

Authorization: Basic xxx（认证信息）

请求体 / 响应

JSON 格式的数据：{ "title": "Hello" }

PUT /apis/api.console.halo.run/v1alpha1/posts/{name}/publish

Headers:
  Accept: */*
  Authorization: Basic bWV0ZXJzcGhlcmV0ZXN0OkZpdDJjbG91ZEAyMDIz

// 这一行 Authorization 就是你的「身份证」
// Basic 后面是 用户名:密码 的 Base64 编码
HTTP 200 OK

    "phase": "PUBLISHED"      // DRAFT → PUBLISHED
  },
  "metadata": {

02 什么是接口测试？

跳过前端界面，直接用 Postman 对后端发 HTTP 请求，验证后端所有环节是否正常。

接口测试覆盖的 4 个验证层
✅ 路由匹配　　URL 对不对？方法是 GET 还是 POST？
✅ 认证拦截　　没带 token 会不会被拦住？
✅ 业务逻辑　　必填字段没填会不会报错？数据处理对不对？
✅ 数据库操作　　数据真的存进去了吗？能查出来吗？
→ 一次 Postman 请求，同时验证 4 层。这就是接口测试的威力。

测试金字塔——接口测试性价比最高

E2E 测试（少量 · 慢 · 全覆盖）
★ 接口测试（中量 · 快 · 高覆盖） ← 你在学的
单元测试（大量 · 最快 · 精确到函数）
03 什么是接口文档？

接口文档是前后端之间的「合同」——规定了每个接口的 URL、参数、返回值。

POST /apis/api.console.halo.run/v1alpha1/posts 文档 3.6 · 编写一篇文章 
// 请求头 ——「我是谁」+「我发的是什么格式」
Authorization: Basic bWV0ZXJzcGhlcmV0ZXN0OkZpdDJjbG91ZEAyMDIz
Content-Type: application/json

// 请求体 —— 文档中标注了哪些是「必填」
{
  "post": {
    "spec": {
      "title": "我的文章",       // ★ 必填
      "slug": "my-post",         // ★ 必填
      "publish": false,          // ★ 必填 默认 false
      "visible": "PUBLIC",       // ★ 必填 可选: PUBLIC/INTERNAL/PRIVATE
      "categories": [],          // 可选
      "tags": []                 // 可选
    }
  },
  "content": {
    "raw": "<p>内容</p>",
    "content": "<p>内容</p>",
    "rawType": "HTML"
  }
}

// 响应 —— 文档告诉你「会返回什么」
→ 200 OK
{
  "status": { "phase": "DRAFT" },
  "metadata": { "name": "abc-123" }  // ★ 后续接口需要用这个 name
}

接口文档中你必须关注的 6 类信息

① 路径 + 方法

往哪发？POST /posts

② 请求头

认证方式、数据格式

③ 参数（必填/选填）

哪些必填？数据类型？取值范围？

④ 响应结果

成功返回什么？字段含义？

⑤ 状态码

200 成功 · 400 参数错 · 401 未认证 · 404 不存在

⑥ 参数关联关系

「metadata 中的 name」—— 接口间的依赖

04 后端如何创建接口？

Halo 用 Java 的 链式路由构建器，在一个文件中定义一组接口。

PostEndpoint.java — 文章接口定义
@Component
public class PostEndpoint implements CustomEndpoint {
 
  RouterFunction endpoint() {
    return SpringdocRouteBuilder.route()
      .GET("posts", this::listPost) // 文章列表
      .POST("posts", this::draftPost) // 创建草稿
      .PUT("posts/{name}/publish", this::publishPost) // 发布
      .PUT("posts/{name}/recycle", this::recyclePost) // 回收
      .DELETE("posts/{name}/content", this::deleteContent)
      .build();
  }
}

07 如何调试并编写接口测试？

文档中的每一个字段、每一个约束，都是一条测试用例的种子。

✅

正向测试

  • 所有必填字段正常填写
  • 全字段创建文章
  • 创建后能查到
  • 创建后能发布
❌

反向测试

  • 不带认证 → 401
  • 缺少必填字段 → 400
  • 请求方法错误 → 405
  • 无效枚举值 → 400
⚠️

边界测试

  • title 为空字符串
  • slug 含特殊字符
  • 重复的 metadata.name
  • 超长标题 10000 字
Postman Tests 断言脚本
// 验证状态码
pm.test("状态码是200", () => {
  pm.response.to.have.status(200);
});
// 验证文章状态
pm.test("文章是草稿", () => {
  const json = pm.response.json();
  pm.expect(json.status.phase).to.eql("DRAFT");
});
// 提取 name 供下个接口使用
pm.environment.set("postName", pm.response.json().metadata.name);
08 完整链路：从写文章到发布

5 个接口按顺序调用，每一步的返回值是下一步的输入。

1

创建文章草稿

POST /posts 文档 3.6 
→ 返回 { metadata: { name: "abc-123" }, status: { phase: "DRAFT" } }
传递 name = "abc-123" 到下一步 ↓
2

发布文章

PUT /posts/{{postName}}/publish 文档 3.7 
→ 返回 { status: { phase: "PUBLISHED" } }
文章已上线，传递 postName 到评论接口 ↓
3

发表评论

POST /comments 文档 3.9 
Body: { subjectRef: { name: "{{postName}}" } }
→ 返回 { metadata: { name: "comment-456" }, spec: { approved: false } }
传递 commentName = "comment-456" ↓
4

审核评论

PUT /comments/{{commentName}} 文档 3.10 
Body: { spec: { approved: true } }
→ 返回 { spec: { approved: true } }
回到文章，使用 postName 进行回收 ↓
5

回收文章

PUT /posts/{{postName}}/recycle 文档 3.8 
→ 返回 { spec: { deleted: true } }
